In a rush? The one-line install is curl -fsSL https://github.com/joym-gits/bugsift/releases/latest/download/install.sh | bash. Everything below explains what that command does, what happens next, and how to recover if anything breaks.

Before you begin

What you're installing

  • Backend - FastAPI + Postgres + Redis + a worker that runs the triage pipeline.
  • Frontend - A Next.js dashboard served at http://localhost:8080.
  • Sandbox - A scoped Docker proxy + per-reproduction ephemeral containers.
  • Everything runs as Docker containers on your machine. Nothing is uploaded anywhere except outbound calls you configure (LLM provider, Slack, Jira).

macOS users: install Docker Desktop, launch it, and wait until the whale icon stops animating before running the installer.

Linux users: make sure your user is in the docker group so docker commands work without sudo: sudo usermod -aG docker $USER && newgrp docker.

Windows users: install Docker Desktop with the WSL2 backend; run everything from an Ubuntu (or similar) WSL shell, not PowerShell.

Pre-flight check

Before you run the installer, paste this one-liner. It should print all OK at the end:

command -v docker >/dev/null && echo "docker: $(docker --version)" \
 && docker compose version >/dev/null && echo "compose: OK" \
 && docker info >/dev/null 2>&1 && echo "daemon: running" \
 && command -v curl >/dev/null && echo "curl: OK" \
 && command -v openssl >/dev/null && echo "openssl: OK" \
 && echo "all OK"

If any line fails, fix that before continuing - the installer will refuse to start otherwise.

Part 1 - Install bugsift

Step 1.1 - Pick where to install

The installer creates a bugsift/ directory in whatever folder you run it from. Choose a location you can find later:

cd ~                    # or wherever you keep services

Step 1.2 - Run the installer

curl -fsSL https://github.com/joym-gits/bugsift/releases/latest/download/install.sh | bash

This takes 30 seconds to 3 minutes depending on your connection (most of the time is pulling Docker images - ~800 MB first time). You'll see output like:

bugsift installer
  target: /home/you/bugsift
  downloading docker-compose.yml
  ✓ docker-compose.yml written
  generating secrets + .env
  ✓ .env written
  pulling images
  ...
  running database migrations
  ...
  starting stack

✓ bugsift is up
  dashboard:       http://localhost:8080
  bootstrap token: TyAakHeOSiq4C_P14l4jrXEYYfWyOAtCcHescirNwyw
  env file:        /home/you/bugsift/.env

Step 1.3 - Save the bootstrap token

Copy the bootstrap token printed at the end. You'll need it in Part 2. It's also saved in ~/bugsift/.env under BUGSIFT_BOOTSTRAP_TOKEN if you forget.

Step 1.4 - Verify everything is running

cd ~/bugsift
docker compose ps

You should see seven containers, all Up:

bugsift-backend-1      Up
bugsift-docker-proxy-1 Up
bugsift-frontend-1     Up
bugsift-nginx-1        Up
bugsift-postgres-1     Up (healthy)
bugsift-redis-1        Up (healthy)
bugsift-worker-1       Up

If any container is Exited or Restarting, jump to Part 12 - Troubleshooting.

Step 1.5 - Open the dashboard

Open http://localhost:8080 in your browser. You should see the bugsift landing page with a "Get started" button.

Running on a remote server? Replace localhost with the server's public IP. If the page doesn't load, check your firewall allows inbound port 8080 (sudo ufw allow 8080 on Ubuntu). Better: jump to Part 10 to set up a real domain with TLS.

Part 2 - Register the GitHub App

bugsift needs its own GitHub App to receive webhooks and post comments. You register it once; bugsift handles everything else (client secret, webhook secret, PEM key) encrypted in its own database.

Step 2.1 - Start onboarding

Click Get started on the landing page, or navigate directly to http://localhost:8080/onboarding. You'll see Step 1 - Register your GitHub App.

Step 2.2 - Make sure you're logged into GitHub

Before clicking Register, open a tab to github.com and confirm you're signed in as the account that will own the App. If you're not signed in, GitHub's manifest flow 500s with a confusing error.

Step 2.3 - Click Register GitHub App

The first click opens a dialog asking for your bootstrap token (the one from Step 1.3 - required when bugsift runs in production mode).

If curl | bash installed with BUGSIFT_ENV=development instead, this step is skipped. That's fine for localhost dev; just click through.

Paste the token and click Register again.

Step 2.4 - Complete GitHub's manifest flow

A new tab opens on github.com/settings/apps/new?state=... with a pre-filled manifest. GitHub asks you to:

  1. Confirm the App name (bugsift auto-adds a suffix if the base name is taken).
  2. Confirm ownership (personal account vs. organisation).
  3. Click Create GitHub App.

GitHub redirects back to bugsift with a temporary code. bugsift exchanges it for the App credentials, encrypts them with your Fernet key, and stores them. The onboarding page auto-advances to Step 2 - Install the App.

The onboarding tab didn't update? GitHub's redirect ended up in the background tab. Switch to it; if it still shows "Waiting on GitHub...", refresh - bugsift's callback has already fired by then.

Part 3 - Sign in and install on a repo

Step 3.1 - Sign in with GitHub

Click Sign in with GitHub at the top-right of the dashboard. Standard OAuth flow: authorise bugsift to read your profile email. You'll be redirected back as a signed-in user.

You are now an admin. The first user on a fresh deployment is automatically granted the admin role. Later users default to triager and can be promoted from the Admin → Users page.

Step 3.2 - Install the App on a repo

From Step 2 of onboarding (or the GitHub settings page at /github):

  1. Click Install on a repo.
  2. GitHub asks which account and which repos to install on.
  3. Pick a test repo first - ideally one you own and that has a few open issues, so you can verify triage without waiting for new ones. If you want bugsift to ingest historical issues for dedup, pick one with at least 20-30 closed issues.
  4. Click Install.

GitHub redirects back to bugsift. The installation is now linked to your user.

Step 3.3 - Verify

Go to Queue (the default landing page for signed-in users). You won't see any cards yet - that's expected. In the sidebar, under Installed repos, the repo you just installed on should appear. Under Onboarding, Steps 1 and 2 should be green-ticked.

Part 4 - Add an LLM API key

bugsift is BYOK (bring your own key). The pipeline only runs against your key, so you choose the provider, pay only for your own usage, and keep the data on keys you trust.

Step 4.1 - Choose a provider

ProviderBest forGet a key
Anthropic (Claude)Default. Highest triage accuracy in our golden set.console.anthropic.com
OpenAI (GPT-4o)Fastest classification.platform.openai.com/api-keys
Google (Gemini)Cheapest per token at scale.aistudio.google.com/app/apikey
OllamaFully local, zero third-party calls, free. Requires a GPU for sensible speed.ollama.com

If you're not sure, pick Anthropic - it's the default model bugsift validates against.

Step 4.2 - Create a key on the provider's side

Follow the provider's docs. For Anthropic specifically:

  1. Visit the API Keys page.
  2. Click Create Key.
  3. Copy the key (starts with sk-ant-api03-...). You won't see it again.
  4. Fund your account or use the free tier - Anthropic gives a small free credit to new accounts.

Step 4.3 - Add the key to bugsift

In the dashboard:

  1. Click Settings in the left sidebar.
  2. Scroll to LLM provider keys.
  3. Pick your provider from the dropdown.
  4. Paste the key.
  5. Click Save.

The key is encrypted at rest using your Fernet key before it's stored in Postgres.

Step 4.4 - Test the key

On the same page, click Test next to the saved key. You should see ✓ looks good · latency 350 ms or similar. If it fails, the error message tells you whether it was a bad key, the wrong endpoint, or a rate-limit issue.

Using Ollama? In .env (in ~/bugsift/.env) set OLLAMA_BASE_URL=http://host.docker.internal:11434 (macOS/Windows) or your host's LAN IP (Linux - host.docker.internal isn't available by default). Restart the backend: docker compose restart backend worker. Then add the key in the dashboard with any placeholder value (Ollama itself doesn't auth).

Part 5 - Configure a repo

Each installed repo can be configured independently.

Step 5.1 - Open the repo settings

On Onboarding → Step 3 (or via GitHub → {your repo} → Configure).

Step 5.2 - Choose a mode

ModeBehaviour
dry-run (default, recommended)Every card lands in the dashboard; you approve, edit, or skip each one. Nothing is posted to GitHub until you click Approve.
autoAllow-listed actions (confirmed duplicates, needs-info replies) execute without approval. Other actions still require approval.

Start with dry-run until you trust the pipeline on your repo.

Step 5.3 - Set a monthly budget

The default is $10/month per repo. When that's hit, cheap steps (classify, draft comment) keep running so cards still land, but expensive steps (dedup, retrieve, reproduce) are skipped. The card shows a "budget exhausted" badge.

Adjust in the same settings panel.

Step 5.4 - Tone + label map (optional)

  • Tone - professional, friendly, or concise. Affects the drafted comment's voice.
  • Label map - translates bugsift's internal labels (bug, needs-info, duplicate, etc.) to the exact label strings your repo uses (kind/bug, status/needs-info, etc.). Skip this until you've seen a few cards to know which labels you actually want.

Part 6 - Your first triage card

Step 6.1 - Open an issue

On the repo you installed bugsift on, open a new issue. You can:

  • Manually create one via github.com.
  • Or wait for a real user-opened issue.
  • Or backfill: GitHub → {repo} → Backfill pulls recent closed issues so there's historical data for dedup to work against.

Step 6.2 - Wait ~90 seconds

bugsift's webhook receives the issue, enqueues a triage job, and the worker runs the pipeline (classify → dedup → retrieve → reproduce → draft). Refresh the Queue page; a card should appear.

Step 6.3 - Read the card

A card tile shows at a glance:

  • Repo · issue number
  • Classification + confidence (e.g. bug 0.91)
  • Severity pill (blocker / high / medium / low)
  • "PII scrubbed" pill if the redactor stripped anything
  • "Learning from N" pill if corrections from past approvals were applied
  • SLA badge if a routing rule set one
  • Relative time since creation

Click the tile. A side sheet opens with the full card:

  • Rationale - why the classifier picked that category.
  • Suspected files - top 3-5 likely-relevant files with line ranges and deep links.
  • Reproduction verdict + log - if a repro was attempted, whether it reproduced and the last lines of output.
  • Duplicates - any prior issues the dedup step flagged.
  • Regression suspects - recent pushes that touched the suspected files.
  • Suggested assignees - from CODEOWNERS (uncheck anyone you don't want assigned).
  • Draft comment - editable before approval.
  • Proposed labels - uncheck any you don't want applied.

Step 6.4 - Act

  • Approve - posts the comment, applies labels, assigns selected users (and optionally creates a Jira ticket).
  • Edit - change the draft, then Approve.
  • Skip - no action on GitHub; the card moves to History with skipped status.
  • Retry - if the pipeline short-circuited (e.g. you hadn't added the LLM key yet), rerun with fresh input.

Every approve / skip / edit writes a correction row - on the next triage run for this repo, the classify and comment prompts include a "Recent operator guidance" block so the pipeline learns from your decisions.

Part 7 - Optional integrations

Step 7.1 - Slack notifications

  1. In Slack, create an Incoming Webhook for the channel you want alerts in. Copy the URL (starts with https://hooks.slack.com/services/...).
  2. In bugsift: Settings → Slack destinations → Add.
  3. Paste the webhook URL, give it a name, pick which events to subscribe to (new_card, approved, regression, sla_breach).
  4. Click Test - a sample message should land in the channel.

Step 7.2 - Jira destinations

  1. In Atlassian, go to id.atlassian.com → Create API token. Copy it.
  2. In bugsift: Settings → Ticket destinations → Add.
  3. Fill in:
    • Site URL - https://your-org.atlassian.net (must be HTTPS, must be a public DNS name - SSRF guard rejects private IPs).
    • Email - your Atlassian account email.
    • API token - the one you just created.
    • Default project key - e.g. ENG.
    • Default issue type - usually Bug.
  4. bugsift validates the credentials against /myself before saving.

Approved feedback-widget cards route to this destination when configured on the feedback app.

Step 7.3 - Feedback widget

bugsift can ingest free-form bug reports from your end-users via an embeddable widget.

  1. Feedback apps → Register a new app.
  2. Pick a name, the target repo, and list the origins the widget will be embedded on (e.g. https://app.example.com). At least one origin is required - empty allowlists are refused.
  3. Save. You get a public_key (prefix pk_...) and an HTML snippet like:
<script src="https://yourbugsift.example.com/api/widget.js"
        data-app-key="pk_abc123" defer></script>
  1. Drop the snippet into your product's HTML. End users see a small "Report a bug" button that opens a modal. Reports land on the bugsift queue, go through the same triage pipeline, and route to either GitHub or Jira when you approve.

Step 7.4 - CODEOWNERS for assignee suggestions

If your repo has a CODEOWNERS file, bugsift parses it and suggests assignees based on the suspected files on each card. Refresh the cache after editing CODEOWNERS:

GitHub → {repo} → Refresh CODEOWNERS

Or wait - bugsift refreshes automatically on every push to the default branch.

Part 8 - Routing rules and SLA

Step 8.1 - Create a rule

Admin → Rules → New rule. Rules run after every card is classified and accumulate in priority order (lower number runs first).

Example rule - "Blocker bugs get paged to the security team":

  • Name: Blocker bugs → sec-team + SLA 60 min
  • Priority: 10
  • Match:
    • Classification: bug
    • Severity: blocker
  • Then (actions):
    • Assign: @sec-engineer-1, @sec-engineer-2
    • Add labels: security, needs-review
    • Notify Slack destination: (pick the #incidents destination)
    • SLA minutes: 60

Click Create. The rule fires on the next card that matches.

Step 8.2 - SLA tracking

Any rule that sets sla_minutes attaches an SLA to matching cards. The SLA watcher (a background task inside the backend) runs every 60 seconds:

  • Before breach: tile shows a grey clock badge with time remaining (SLA in 42m). Amber when under 10% remaining.
  • After breach: tile turns red (breached 14m ago), Slack fires a sla_breach event, audit log records sla.breached.

The Admin → Metrics page shows SLA compliance % across the selected time window.

Part 9 - Admin features

Admin-only pages live in the Admin section of the sidebar.

Step 9.1 - Users

  • List every user who's signed in on this deployment.
  • Change roles from the dropdown (admin, triager, viewer).
  • The last admin cannot demote themselves - promote someone else first.

Step 9.2 - Audit log

Every security-relevant action is recorded append-only:

  • User login, role changes.
  • Card approve / skip / edit.
  • API key create / delete.
  • GitHub App register / delete, install linked.
  • Rule / destination create / update / delete.
  • SLA breach.

Filter by actor, action type, or target type. Export CSV dumps the matching rows for SOC 2 auditors or your compliance team.

Step 9.3 - Metrics

A single-page dashboard with a 7 / 14 / 30 / 90-day window selector:

  • Cards created per day (sparkline).
  • LLM spend per day (sparkline).
  • Cost by provider / model / pipeline step.
  • Outcome mix (posted / skipped / pending) + approval rate.
  • PII scrub rate.
  • SLA compliance %.
  • Classification + severity breakdowns.

Part 10 - Deploying beyond localhost

Localhost is fine for evaluating and for solo use. For a team deployment:

Step 10.1 - Pick a host

Any Linux VM with Docker works. Good options:

HostCheapest tierGood for
Hetzner CloudCX21 · 2 vCPU · 4 GB · ~€5/moEU teams
DigitalOceanBasic 2 GB · $12/moGlobal coverage
LinodeNanode 2 GB · $12/moGlobal coverage
AWS Lightsail2 GB · $10/moAlready in AWS
GCP e2-small~$13/moAlready in GCP

Any 2 vCPU / 2 GB / 20 GB VM running Ubuntu 22.04+ or Debian 12+ works.

Step 10.2 - Prepare the server

# SSH in
ssh root@your-server-ip

# Install Docker (Ubuntu/Debian)
curl -fsSL https://get.docker.com | sh
usermod -aG docker ubuntu   # or your non-root user

# Open the firewall (if ufw is active)
ufw allow 80
ufw allow 443

Step 10.3 - Point a domain at it

In your DNS provider, create an A record pointing bugs.example.com (or whatever subdomain you want) at the server's public IPv4. Wait for propagation (2-30 min typically).

Step 10.4 - Install bugsift

Same command as localhost:

curl -fsSL https://github.com/joym-gits/bugsift/releases/latest/download/install.sh | bash
cd ~/bugsift

Step 10.5 - Apply the Caddy TLS overlay

Caddy provisions Let's Encrypt certificates automatically. Add your domain + email to .env:

cat >> .env <<EOF
BUGSIFT_DOMAIN=bugs.example.com
BUGSIFT_ACME_EMAIL=ops@example.com
BUGSIFT_PUBLIC_URL=https://bugs.example.com
NEXT_PUBLIC_API_BASE_URL=https://bugs.example.com/api
EOF

Download the Caddy overlay and restart with it merged in:

curl -fsSLo docker-compose.caddy.yml \
  https://raw.githubusercontent.com/joym-gits/bugsift/main/deploy/docker-compose.caddy.yml

docker compose \
  -f docker-compose.yml \
  -f docker-compose.caddy.yml up -d

The first boot requests a certificate from Let's Encrypt (watch with docker compose logs -f caddy). Once issued, https://bugs.example.com serves the dashboard with a valid TLS cert.

Step 10.6 - Update the GitHub App webhook URL

Because your public URL changed, the GitHub App's webhook endpoint needs updating:

  1. Go to https://github.com/settings/apps/your-app-name/permissions.
  2. Under Webhook URL, update to https://bugs.example.com/api/webhooks/github.
  3. Save.

New webhooks land directly - no more smee.io forwarder needed.

Part 11 - Day-to-day operations

Always run these from inside ~/bugsift/:

cd ~/bugsift

View logs

docker compose logs -f backend          # follow backend logs
docker compose logs -f worker           # follow worker logs
docker compose logs --tail 100 nginx    # last 100 lines of nginx
docker compose logs                     # everything (noisy)

Restart a service

docker compose restart backend
docker compose restart worker

Stop / start the whole stack

docker compose down        # stops containers; volumes (data) survive
docker compose up -d       # starts everything

Upgrade to the latest release

docker compose pull
docker compose run --rm backend alembic upgrade head
docker compose up -d

To pin a specific version, set BUGSIFT_IMAGE_TAG=v0.1.0 in .env before pulling.

Back up your data

Everything worth saving lives in two volumes: bugsift_postgres_data (cards, corrections, audit log, encrypted keys) and bugsift_redis_data (queue state, which is fine to lose).

Nightly pg_dump is usually enough:

docker compose exec -T postgres pg_dump -U bugsift bugsift \
  | gzip > "backup-$(date +%F).sql.gz"

Back up .env too. It contains the Fernet key - lose it and every encrypted row (API keys, App PEM, Slack URLs, Jira tokens) is unrecoverable. Store .env in a password manager or encrypted backup.

Restore from backup

gunzip -c backup-YYYY-MM-DD.sql.gz \
  | docker compose exec -T postgres psql -U bugsift bugsift

Restore .env to the same content (or at least the same BUGSIFT_ENCRYPTION_KEY) before booting, or decryption fails.

Part 12 - Troubleshooting

Installer fails with docker: command not found

Install Docker first. See Before you begin.

Installer fails with docker daemon is not running

  • macOS/Windows: open Docker Desktop and wait for the whale icon to stop animating.
  • Linux: sudo systemctl start docker then sudo systemctl enable docker.

Installer fails with permission denied on the Docker socket

Your user isn't in the docker group:

sudo usermod -aG docker $USER
newgrp docker
# Then re-run the installer.

Port 8080 already in use

Something else is bound to 8080. Either stop it or change bugsift's port:

cd ~/bugsift
echo "BUGSIFT_PUBLIC_PORT=18080" >> .env
docker compose up -d nginx
# Now use http://localhost:18080

If you change the port, also update BUGSIFT_PUBLIC_URL and NEXT_PUBLIC_API_BASE_URL in .env to match, then docker compose up -d everything.

"Bad gateway" on sign-in

Backend restarted since nginx resolved its IP. bugsift's nginx config uses a Docker DNS resolver that should auto-recover; if not:

cd ~/bugsift
docker compose restart nginx

If it persists, check docker compose logs backend - the backend might be crashing. A common culprit is a malformed BUGSIFT_ENCRYPTION_KEY in .env. Regenerate it:

python3 -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
# Paste the output into .env and docker compose restart backend

Warning: rotating the encryption key invalidates every encrypted row in the database. If you had saved LLM keys / Slack URLs / Jira tokens, you'll have to re-enter them.

Onboarding says "Bootstrap token required" but I set one

The backend only reads .env on startup. After changing BUGSIFT_BOOTSTRAP_TOKEN:

docker compose restart backend

Then try again - paste the current value from ~/bugsift/.env.

I opened an issue but no card appeared

Walk through these in order:

  1. Is triage enqueued? docker compose logs backend | grep webhook - you should see webhook received issues.opened. No line = GitHub never reached you. Check that the App is installed on the repo and that webhooks are delivering (GitHub → {App} → Advanced → Recent Deliveries).
  2. Is the worker picking it up? docker compose logs worker | grep triage - you should see processing issue_opened. No line = worker isn't running or Redis is unreachable.
  3. Is the LLM call failing? docker compose logs worker | grep -i error - a 401 / 429 here means your key is wrong or rate-limited.
  4. Is dashboards frontend up but the backend is not? Hit http://localhost:8080/api/health directly. It should return {"status":"ok","version":"0.x.x"}.

LLM key test returns 401

Copy your key fresh from the provider (Anthropic keys start with sk-ant-, OpenAI with sk-proj- or sk-, Google with AIza). Common mistakes:

  • Trailing whitespace from the copy-paste.
  • Using the org admin key instead of a user key.
  • The key is rate-limited - try again in 60 s.

Webhook not arriving on localhost

The installer auto-provisions a smee.io channel that forwards to backend:8000/webhooks/github. If it's not working:

docker compose logs backend | grep smee

You should see smee forwarder starting: https://smee.io/XXX. If you see connection errors, either smee.io is having an outage (rare) or your server has no outbound HTTPS. Test:

curl -I https://smee.io

If that hangs, your network blocks outbound HTTPS to smee.io - expose the port directly or put a domain in front and set a real webhook URL in the GitHub App settings.

Migration failure on upgrade

If alembic upgrade head errors out, the migration is usually safe to retry after fixing the root cause. Check the error message; common cases:

  • Duplicate column - a previous upgrade was interrupted. Jump to head with alembic stamp head if you're sure the schema is already there (risky - ideally restore from backup first).
  • Missing table - the database wasn't seeded. docker compose down && docker compose up -d and retry.

"No LLM key configured" on cards that existed before I added a key

The pipeline checked your key at triage time and there was none, so the card is pending with no classification. Click Retry on the card - the orchestrator reruns with your current key.

Container killed / OOM

2 GB RAM is the minimum and leaves little headroom. Check:

docker stats --no-stream

If Postgres is eating > 1 GB, reduce its shared_buffers (uncommon). If the worker is the culprit, you're probably processing a large backfill job - wait for it to finish.

Everything else - ask

  • GitHub issue: github.com/joym-gits/bugsift/issues - include the output of docker compose logs --tail 200 (redact your .env values first).
  • Twitter: @_joymukerjee - for "is this even working for anyone else" style pings.

Part 13 - Uninstall and cleanup

To stop bugsift without deleting anything:

cd ~/bugsift
docker compose down

Volumes survive - you can docker compose up -d again and everything picks up where it left off.

To delete everything including your data:

cd ~/bugsift
docker compose down -v            # -v deletes volumes (postgres + redis data)
docker image rm $(docker images -q 'ghcr.io/joym-gits/bugsift-*')
cd ..
rm -rf bugsift

To also revoke the GitHub App:

  1. Go to github.com/settings/apps.
  2. Find your bugsift App.
  3. Click Delete App at the bottom.

Any repos the App was installed on lose their webhook connection automatically.
