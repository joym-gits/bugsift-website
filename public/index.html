The problem

Reading every issue is the work. The actual engineering is blocked behind it.

At 100 open issues on a solo repo, the maintainer spends more time sorting signal from noise than fixing bugs. Most existing tools either overreach (auto-generating PRs the maintainer never asked for) or give up on privacy (shipping your issues to a third-party SaaS that indexes them forever).

What other tools offer

  • Auto-PR bots that write code you never reviewed and open pull requests with no context.
  • Hosted SaaS triage that pipes every issue body and GitHub token through someone else's cluster.
  • Chat-based agents that require you to prompt them back and forth for each issue.
  • Static classifiers that are as good on day 90 as day 1 — they never learn your team's voice.

What bugsift offers instead

  • Triage-only scope. Classify, dedup, reproduce, draft. No code changes, no PRs, no overreach.
  • Self-hosted by design. Runs on your box. Tokens and issue bodies never leave your infrastructure.
  • Human-in-the-loop by default. Every draft lands in a dashboard. You approve, edit, or skip.
  • Learns from corrections. Every override feeds back into the next run so triage compounds with use.
How it works

One deterministic pipeline. Five steps. ~90 seconds per issue.

No LangChain, no ReAct loops, no autonomous planning. A ~300-line orchestrator runs the steps in a fixed order. You can read every prompt and every decision in the dashboard.

1

Classify

bug · feature-request · question · docs · spam · other — with confidence. PII redacted before the prompt.

2

Dedup

Cosine search over past-issue embeddings + LLM judge. Short-circuits on true duplicates and drafts a "duplicates #123" reply.

3

Retrieve

Points at the 3–5 most likely relevant files — with line ranges and deep links. Refined post-repro by traceback paths.

4

Reproduce

Drafts a minimal repro, runs it in a hardened ephemeral sandbox (network off, caps dropped, 60-second kill).

5

Draft

Writes the triage comment, proposes labels, picks an action. Lands on a card in the dashboard for your approval.

Hardened by default

The security posture you'd expect before a paid customer asks for it.

Every item below is on by default. You opt out, not in. Full security audit + mitigation report lives in the repo.

✓

Sandbox isolation

Reproduction containers run with network_mode=none, read-only root, tmpfs scratch, every Linux capability dropped, and a 60-second kill.

✓

No raw docker.sock on app containers

The worker talks to Docker through a scoped socket proxy (containers + images only). The real socket is mounted read-only on the proxy, nowhere else.

✓

Every secret Fernet-encrypted at rest

GitHub App PEM, OAuth client secret, webhook secret, your LLM keys, Slack webhook URLs, Jira API tokens — all encrypted with a master key that never ships in logs.

✓

CSP with per-request nonces

Next.js middleware emits script-src 'self' 'nonce-...' 'strict-dynamic' so injected scripts without the nonce are refused by the browser.

✓

SSRF-safe outbound HTTP

Every user-supplied URL (Jira site, etc.) resolves and is rejected if it points at a private, loopback, link-local, or CGNAT address.

✓

PII redaction before every LLM call

Emails, phones, SSN, Luhn-valid cards, provider API keys, JWTs, URL credentials — scrubbed with stable hashed tokens so the LLM never sees sensitive data.

✓

Install-callback CSRF protected

GitHub install callback verifies installation ownership against the authenticated login. Can't re-parent an installation via a crafted link.

✓

Append-only audit trail

Every login, role change, approve, key rotation, and App register/delete is recorded — CSV-exportable for SOC 2 auditors or your compliance team.

Get started

Install in one command. Everything else is clicks.

Requires Docker 24+ and ~2 GB RAM. Runs on any Linux box, macOS laptop, or cloud VM. No cloning, no .env editing, no manual migrations.

$ curl -fsSL https://github.com/joym-gits/bugsift/releases/latest/download/install.sh | bash
1
The installer runs Verifies Docker, generates a Fernet key + session secret + bootstrap token, writes .env, pulls pre-built images from GitHub Container Registry, runs migrations.
2
Open the dashboard The installer prints your URL and the first-run bootstrap token. Visit the dashboard, click Register GitHub App, paste the token.
3
Install on a repo GitHub's manifest flow stores credentials in bugsift's encrypted DB. Install the App on whichever repo you want triaged.
4
Add an LLM key Paste your Anthropic, OpenAI, Google, or Ollama key in Settings. Open an issue on the installed repo. A triage card appears within ~90 seconds.

Want to pin a specific release for reproducibility? Every release publishes install.sh and docker-compose.prod.yml — use BUGSIFT_IMAGE_TAG=v0.1.0 to lock your deployment. Full self-host guide including backups and TLS via Caddy: deploy/README.md.

FAQ

Things people ask in the first five minutes.

Do my issue bodies get sent to OpenAI or Anthropic?

Only if you configure a provider and add a key. Before any prompt leaves the server, the PII redactor scrubs emails, phones, SSN, credit cards, provider API keys, JWTs, and URLs-with-credentials, replacing them with stable hashed tokens. The LLM never sees raw sensitive data. You can also run fully local with Ollama and no third-party calls at all.

Can I use it for private / commercial repositories?

Yes. bugsift runs on your infrastructure. Your GitHub tokens, issue bodies, and operator decisions never touch a third-party SaaS. The licence is Apache 2.0 — use it for anything, commercial or otherwise.

What does it cost?

bugsift itself is free and open-source (Apache 2.0). You pay only for the LLM tokens your configured provider charges. Budget caps with graceful degradation ship in the box — when the monthly cap is reached, cheap steps (classify, comment) still run so cards still land, while expensive steps (dedup, retrieve, reproduce) are skipped and the card is flagged.

Do I need a cloud account?

No. A laptop with Docker works. A $5/mo VPS with 2 GB RAM works. Your existing Kubernetes cluster works. Everything ships as a single docker-compose stack. There's no bugsift.com instance you need an account on.

What happens when GitHub sends a webhook? Can my localhost receive it?

The installer auto-provisions a smee.io channel and runs an in-process forwarder, so GitHub can reach your localhost deployment without you exposing any ports. When you deploy behind a real domain, the Caddy overlay handles TLS via Let's Encrypt and webhooks land directly.

Will it write code or open pull requests?

No. That's a deliberate non-goal. bugsift is triage-only: classify, dedup, reproduce, draft a comment, propose labels, suggest assignees. Every action lands on a card in the dashboard for you to approve, edit, or skip. The maintainer is always in the loop.

How does it get better over time?

Every approve / skip / edit writes a correction row — the delta between what the pipeline suggested and what you actually did. The next triage run on the same repo reads the most recent corrections and injects them into the classify and comment prompts as "operator guidance." So triage decision #100 compounds every override you made on decisions #1–99.

Can I customise the prompts or pipeline?

Yes. The orchestrator lives in backend/src/bugsift/agent/ and is deliberately ~300 lines of straight Python — no framework, no magic. Prompts are Jinja2 templates in agent/prompts/. Fork, edit, rebuild, ship your own images.

How do I upgrade?

docker compose pull && docker compose run --rm backend alembic upgrade head && docker compose up -d. Migrations are backward-compatible within a minor version. Major releases list operator-action items in the release notes.
